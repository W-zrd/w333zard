Bug Hunting XSS Vulnerability in Jember University Journal Web

Exploiting web for fun & profit (braindead edition)

Introduction

"Your session has been hijacked."

These words send chills down the spine of any cybersecurity professional. But what if I told you I discovered a way to make these words appear on any student's or professor's screen at X University? What if I could access their private messages, grades, or personal information - all because of a simple flaw in how the university's website handles certain requests?

"I have nothing to hide, so why should I care about privacy?" I hear this often, but let's talk a bit more about this kid.. Imagine a scenario where someone could silently monitor your every move on the university's platforms. They could read your scholarship applications, or even alter your academic records. Even more concerning, they could impersonate you, sending messages to your professors or classmates that appear to come directly from you.

Each of those record leaves a digital trace, and each trace contains pieces of your identity. It's about your future. Every grade, every submission, every financial transaction builds a record that could affect your job prospects or anything beyond that. When this information is compromised, it's not just data that's at risk – it's your opportunities, your reputation, and your future.

Recently, while conducting a bug hunting activity of Jember University's web application, I discovered a security vulnerability that could allow attackers to steal users' sensitive information. No alarms, no red flags, no network log or history, just me and their data. Maybe this findings is also a potential backdoor of "Slot Gacor" web defacement in this university

Proof of Concept: Reflected XSS via Host Header Injection

During a recent security assessment on X University’s OJS (Open Journal Systems) platform, I identified a Reflected Cross-Site Scripting (XSS) via Host Header Injection. This vulnerability could allow attackers to inject malicious JavaScript code into the application. The key danger here is that once the malicious script executes in the user’s browser, it can access sensitive data like cookies, session information, or even alter the page content.

Host Header Injection occurs when an application or a backend server trusts and uses the "Host" or "X-Forwarded-Host" HTTP headers without validation. These headers indicate the domain or IP address of the server handling the request. If not properly validated, attackers can manipulate these headers to inject scripts or other malicious content into the response.

Below, I’ll outline the steps to reproduce this finding, followed by a detailed breakdown of the vulnerability’s impact.

  • Intercept any HTTP request to the OJS platform. In this case, i did a simple GET request to the root web page
  • Modify the X-Forwarded-Host Header: Add a custom header to the request with an injected JavaScript payload. Here’s an example of a malicious payload: 
    X-Forwarded-Host: foo"><script>alert(document.cookie)</script><x=".com
  • Send the modified request. If the application is vulnerable, the response from the server will reflect the injected script without sanitization.
The payload is executed by the browser as though it were a legitimate part of the site’s code. In our case, the payload used ("><script>alert(document.cookie)</script>) successfully triggered a pop-up showing the contents of session tokens and other sensitive data. This shows that the attacker could access the user’s cookies. The vulnerability lies in the server's inability to differentiate between safe, trusted content and potentially malicious input from user-controlled headers like X-Forwarded-Host . Since the server lacks proper validation here, any untrusted data from this header can flow freely into the web response.

Impact & Findings

The XSS vulnerability i found significant implications. It can leads to Browser Exploitation. This type of attack can manipulate various functionalities within the victim’s browser. By exploiting this flaw, an attacker can potentially retrieve sensitive data about the user, such as their IP address, which reveals approximate location.

Furthermore, the attacker could trigger the browser to redirect users to external malicious sites without their knowledge. This capability presents a substantial risk of phishing attacks, where victims may be tricked into providing sensitive credentials, such as university login details, to a page that looks identical to the legitimate OJS platform. These redirects can also be used to deploy malware or spyware, further compromising user data and device security.

Beyond data leakage and redirection, attackers can manipulate the browser to initiate other unwanted actions, like automatically placing a phone call if the browser has the necessary permissions. As shown on the image, my payload has successfully trigger the browser to open GSConnect which is an app to open call in Ubuntu. To be honest i could take another lateral movement by leveraging this vulnerability, but i don't. This is enough to confirm that this website is vulnerable to XSS

Conclusion

As we conclude this deep dive into the vulnerability I discovered at Jember University, let's take a moment to reflect our journey. The host header injection vulnerability I found might seem like a technical glitch to some, but it represents a much larger issue that affects us all.

Security isn't just the responsibility of IT departments or security researchers. It's a shared responsibility that requires awareness and action from all of us. Simple habits like using strong passwords, enabling 2FA can make a significant difference.

As we move forward, let this discovery serve as a wake-up call. Your privacy matters. Your data matters. Your digital identity matters. The bad news is this vulnerability has not been fixed, so prepare for the worst. The good news is you have a power to protect yourself. Think critically, Question every damn thing, and trust nobody, because nothing secure when you enter the internet
